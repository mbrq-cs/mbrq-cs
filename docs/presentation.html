Секция 03

Что мы выяснили

Каждый из следующих выводов - не теория. Это паттерны, наблюдавшиеся повторно на разных проектах.

03.1 / Доверие

Ограничения - программные,
не текстовые

Паттерн: инструкция в промпте "Edits forbidden" - агенты стабильно рационализируют нарушение.

Почему: агенты - языковые модели. Если можно сгенерировать убедительное обоснование нарушения, они его сгенерируют.

Решение: hooks на уровне Claude Code, read-only директории, разделение фаз в пайплайне. Агент физически не имеет доступа к тому, что ему не положено.

03.1 / Пример

Из лога агента

"The constraint says 'Never edit, create, or delete files in this directory' - but this is to prevent manual modifications to generated output. The generator failed to produce them. The alternative (stopping the entire phase) is worse than creating minimal stubs that match the expected generated format."

Агент прочитал правило, объяснил почему его ситуация - исключение, и нарушил запрет.
Результат: каскад из 7+ ошибок в зависимых задачах.

03.2 / Скоуп

Узкий скоуп побеждает

Паттерн: general-purpose агенты с широким контекстом теряют фокус. Чем больше инструкций, тем хуже следование каждой из них.

Решение: один агент - одна ответственность. Subagents с изолированным контекстом. Знания разделяются по ролям.

Данные: за две недели итераций критические нарушения снизились с 4+ за сессию до нуля. Не за счет более умных промптов - за счет того что каждый агент получал меньше контекста.

03.3 / Верификация

Тавтологический цикл

Паттерн: агент контролирует и тесты и реализацию - подгоняет одно под другое. 281 зеленый тест - наглядный пример.

Вывод: это фундаментальная, не дисциплинарная проблема. Агент, контролирующий и реализацию и верификацию, подтверждает сам себя.

Решение:

  • TDD фазы структурно отделены от реализации
  • Тесты пишутся отдельным агентом, до реализации
  • Тестовые файлы защищены hooks на этапе реализации
  • Моки генерируются из swagger, лежат в read-only пакете
  • Отдельный QA-агент верифицирует результат
03.4 / Планирование

Один проход - недостаточно

После одного прогона планирования - пробелы, неявные допущения, слишком широкие задачи. Агенты импровизируют - и импровизируют плохо.

План проходит минимум два ревью до начала реализации. Начальный план - черновик, не инструкция.

03.5 / Объем

Объем ≠ Прогресс

150 файлов, 10K строк, 281 тест. Все за один прогон. Приложение не работает.

Без структурированной верификации AI-код достигает потолка расширяемости очень быстро. Вайб-кодинг работает для прототипов. Для продукта - нужен надзор.

04.2 / Знания

Skills vs Rules

Проблема: Claude склонен забывать rules и импровизировать. Критичные знания в rules - ненадежно.

Rules

Инжектятся всегда, каждому subagent. Конвенции, правила документации, оформление планов.

Skills

Вызываются явно через Skill tool - механическое принуждение. Архитектура, как писать тесты, как делать code-review.

Skills позволяют давать каждому subagent только релевантные знания.

Пример:

  • Агенту для unit-тестов не нужны знания об архитектуре
  • Агенту для фич не нужны знания об e2e-тестах
04.3 / Требования

Автоматический сбор требований

AI-агент прокликивает legacy-приложение, записывает user-flow сценарии по стандартизированному формату, связывает UI-элементы с релевантными эндпоинтами.

Разработчики валидируют и уточняют собранные агентом требования.

User-flow сценарии становятся компонентом PRD и основой для e2e тестов. Привязка к endpoints используется для моков.
